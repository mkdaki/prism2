## 1. テスト目的

**推移比較機能を含むPoC**が、以下の観点で「使える」かを自社で検証する：

1. **実用性**: 想定ユースケースで実際に使えるか
2. **価値**: 既存の作業（Excel、手作業）よりも楽になるか
3. **継続利用意向**: 定期的に使い続けたいと思えるか
4. **収益化可能性**: 将来的にお金を払う価値があると感じるか

### 重要：推移比較機能の位置づけ

**推移比較機能（同一対象の継続スクレイピングデータの変化分析）は、Prismの最重要機能である**。

- 単発分析はExcelでも可能
- **時系列での変化検知・トレンド分析こそが真の差別化要因**
- 継続利用（サブスク化）の根拠になる

このため、**推移比較機能の実装完了をE2テストの前提条件**とする。

## 2. 自社テストで検証するペルソナ

E2テストは**自社のみ**で実施する。以下のペルソナを自分自身で演じ、実際の業務データで検証する。

### ペルソナA: データ販売者（自社）
- **職業**: フリーランスのデータ収集・分析者
- **課題**: スクレイピングしたCSVを手作業で集計・レポート化している
- **期待**: 分析コメントを自動生成して納品時間を短縮したい
- **自社での検証方法**: 実際のスクレイピング案件データ（匿名化）を使用

### ペルソナB: せどり実践者（自社）
- **職業**: 会社員（副業でせどり）
- **課題**: Amazon/メルカリの価格差を見つけるのに時間がかかる
- **期待**: 異常値や急変トレンドを素早く把握したい
- **自社での検証方法**: 架空または公開ECデータで価格分析を試す

### ペルソナC: 定期スクレイピング実践者（自社・最重要）
- **職業**: 投資家、事業探索者、マーケットリサーチャー
- **課題**: 定期スクレイピングしているが、変化に気づきにくい
- **期待**: 前回データとの比較、異常値の自動検知、トレンド把握
- **具体的な利用シーン**:
  - 不動産価格の推移監視
  - 競合商品の価格変動追跡
  - 求人市場のトレンド変化分析
  - ECサイトの在庫状況の変化検知
- **自社での検証方法**: 上記いずれかの対象を実際に定期スクレイピングし、推移比較機能を使う

**注**: ペルソナCの検証が最も重要。推移比較機能の価値を実感できるかが成否を分ける。

## 4. テストシナリオ

### シナリオ1: データ販売者の納品業務（ペルソナA向け）

**前提条件**:
- Playwrightでスクレイピングした求人データCSV（100-500行程度）
- クライアントへ分析レポートを納品する想定

**テスト手順**:
1. CSVファイルをPrismにアップロード
2. 統計情報を確認（数値カラムの分布、頻出カテゴリ）
3. LLM分析結果を確認
4. 分析結果をコピーして、簡易レポートを作成（メモ帳/Wordなど）
5. グラフをスクリーンショット取得

**成功基準**:
- アップロードから分析結果確認まで **3分以内**
- 分析コメントが「そのまま使える」または「軽微な修正で使える」
- エラーやフリーズが発生しない

**観察ポイント**:
- どの機能を最初に使うか
- 統計情報とLLM分析、どちらを重視するか
- エクスポート機能があったら使うか

---

### シナリオ2: せどり実践者の価格差分析（ペルソナB向け）

**前提条件**:
- Amazon/メルカリから取得した商品価格データCSV（50-200行程度）
- 商品名、カテゴリ、価格、出品日などを含む

**テスト手順**:
1. CSVファイルをPrismにアップロード
2. 価格カラムの統計情報を確認（min/max/avg）
3. 異常値（極端に安い/高い商品）を探す
4. カテゴリ別の傾向をLLM分析から読み取る
5. 次のアクション（仕入れ判断）に繋がる情報が得られたか評価

**成功基準**:
- **高速性**: アップロードから結果確認まで **2分以内**
- **発見性**: 異常値や注目すべき商品を見つけられる
- **実用性**: 実際の仕入れ判断に使えると感じる

**観察ポイント**:
- 統計情報のどの指標を見るか（min/max/avg/頻出値）
- グラフが役立つか
- 「前回データとの比較」機能が欲しいか

---

### シナリオ3: 定期スクレイピングデータの推移比較（ペルソナC向け・最重要）

**前提条件**:
- **推移比較機能が実装済みであること（E2テストの必須前提）**
- 同一対象を週2-3回、2週間継続してスクレイピング

**想定する具体的な利用シーン**（以下のいずれかで検証）:
1. **不動産価格の推移**: 特定エリアの賃貸・売買物件の価格変動を定期監視
2. **競合商品の価格変動**: Amazon/楽天などのEC上での競合商品の価格推移を追跡
3. **求人市場のトレンド変化**: 特定職種・エリアの求人件数、給与水準の変化を把握
4. **ECサイトの在庫状況の変化**: 特定商品カテゴリの在庫数、品揃えの変化を監視

これらはすべて「定期監視」が必要な領域であり、推移比較機能の真価が発揮される場面。

**テスト手順**:
1. 初回データをアップロード → 基準データとして設定
2. 3日後、同じ対象を再スクレイピング → 2回目データをアップロード
3. **推移比較機能を使用**：2つのデータセットを比較対象として指定
4. 統計情報の差分を確認（平均価格の変化、件数の増減など）
5. **LLM推移分析結果を確認**：「前回と比べて何が変わったか」を自動生成
6. 1週間後、3回目のデータで同様の比較を実施
7. 時系列での変化トレンドを把握できるか確認

**成功基準**:
- 推移比較機能で **2つのデータセット比較が3分以内** に完了
- 統計情報の差分が視覚的にわかりやすい
- **LLMが「変化の要点」を的確に指摘できる**（例：「平均価格が5%上昇」「新規カテゴリが3件出現」）
- 継続的な監視に使いたいと感じる

**観察ポイント**:
- **推移比較機能が「キラー機能」として成立するか**
- どの変化（価格、件数、カテゴリ、異常値）に最も価値を感じるか
- 週次・月次での定期監視に使い続けたいと思えるか
- この機能があればお金を払ってでも使いたいと思えるか

## 8. 成功基準（Go/No-Go判断）

### Goの基準（本格開発へ進む）

以下の **4つすべて** を満たす場合、フェーズFへ進む：

1. **推移比較機能の価値**: 「これこそが求めていた機能だ」と実感できる
2. **時間短縮**: 従来手法と比べて **30%以上の時間短縮** を実感
3. **継続利用意向**: 自分自身が **実際の業務で使い続けたい** と思える
4. **支払い意向**: もし有料サービスなら **月額1,000円以上払ってもよい** と思える
5. **技術的成立**: 大きな技術的問題（パフォーマンス、バグ等）がない

**最重要判断基準**: **推移比較機能が「キラー機能」として成立したか**

### Pivotの基準（方向転換）

以下のような場合、ターゲット顧客や機能を見直す：

- 単発分析は便利だが、推移比較はそこまで重要でない → ターゲット顧客を再検討
- 統計情報は好評だが、LLM分析が不要 → LLMなしの軽量版を検討
- ペルソナA, Bは刺さらないが、ペルソナCには強く刺さる → ターゲットを絞る
- 推移比較は良いが、手動選択が面倒 → 自動比較機能を優先実装

### No-Goの基準（中止またはロングスリープ）

以下の場合、PoC止まりとし、知見を次に活かす：

- **推移比較機能を使っても「手作業の方が良い」と感じる**
- 技術的に解決困難な問題が多発（LLMコスト、パフォーマンス等）
- 自分自身が使い続けたいと思えない
- 支払い意向が **月額500円未満**（事業化困難）

**注**: 自社テストのため、判断基準は「自分の実感」が中心。数値目標は参考値として扱う。

## 9. リスクと対策

### リスク1: 推移比較機能の実装が遅延する
- **影響**: E2テストの開始が遅れる
- **対策**: 推移比較機能の実装スコープを最小限に絞る（MVP）
- **判断基準**: 2週間以上遅延する場合、E2テストを2段階に分ける

### リスク2: 定期スクレイピング対象が見つからない
- **影響**: 推移比較機能の検証ができない
- **対策**: 事前に対象候補をリストアップ（不動産、中古車、EC価格など）
- **代替案**: サンプルデータで時系列データを疑似生成

### リスク3: 推移比較機能が期待外れ
- **影響**: 最重要機能が価値を生まない → Pivot or No-Go
- **対策**: これは検証すべきリスクなので、対策不要（結果を受け入れる）
- **判断**: 期待外れでも、その知見自体が成果

### リスク4: LLMコストが想定以上に高い
- **影響**: 事業化が困難になる
- **対策**: テスト期間中のLLM API使用量・コストを記録
- **判断材料**: 1分析あたりのコストを算出し、収益化の可能性を評価

### リスク5: 技術的な問題（パフォーマンス、バグ）
- **影響**: テスト継続が困難になる
- **対策**: 重大な問題が発生した場合、修正を優先（テスト中断もあり得る）
- **判断**: 修正不可能な問題なら、No-Go判断
